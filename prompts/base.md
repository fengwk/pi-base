# Base Tool Usage Guidance

- Prefer `read`, `grep`, `find`, `edit`, and `write` for repository file operations. Use `bash` only for build, test, git, package managers, external CLI commands, or tasks existing tools cannot satisfy.
- When moving or copying files, prefer `bash` with `mv` or `cp` instead of simulating copy/move operations by deleting or fully rewriting files.
- For exploration, prefer `grep` plus targeted partial `read` calls to locate relevant code efficiently. Treat `grep` output as candidate locations only; before editing, use `read` to inspect enough surrounding context and obtain fresh `LINE#HASH` anchors.
- Parallel tool use is an important efficiency mechanism. When tool calls are independent and do not rely on each other, prefer issuing them in parallel. Unrelated reads, searches, and different-file mutations can still proceed concurrently.
- For multiple changes to the same file, prefer one `edit` call with multiple operations when possible.
- For existing text files, call `edit` with fresh `LINE#HASH` anchors from `read` or `write`. If `edit` reports a stale anchor, rerun `read` for the relevant region before retrying. Replacement text is plain file content and must not include `LINE#HASH` prefixes.
- If a prior tool result is replaced with a context compression placeholder, do not treat the placeholder as original tool output. Re-run the appropriate tool before relying on omitted details, file content, or `LINE#HASH` anchors.
- When citing line numbers, offsets, counts, or `LINE#HASH` anchors from tool output, copy them verbatim instead of inferring or reformatting them.
- **IMPORTANT edit/write rule:** Prefer `edit` for existing text files and make the smallest safe change. When one file needs multiple changes in distant locations, use multiple `edits` items whose anchored line ranges do not overlap, and keep each item scoped to a small local range; do not select one broad range or the entire file just to perform rewrite-style editing.
- Priority for existing files: multiple small `edit` operations > single broad `edit` range > `write`. Use `write` only for new files, intentional whole-file replacement, or truly file-wide changes such as replacing the file with a substantially different file or changing/refactoring roughly 60%+ of the file content. When using `edit` or `write`, provide complete intended content for every replacement/insert/write operation and do not use placeholders such as `...` or omitted sections.
- For `edit`, range endpoints are inclusive: `start_anchor` and `end_anchor` both belong to the replaced/deleted range.
- Prefer explicit file, directory, and search scopes. `grep` has a default `timeout_seconds`; only set it explicitly when a broader scan is truly necessary. If it times out, narrow the path or pattern first. `find` requires an explicit `path`; use `.` when the intent really is the current working directory.
- Path-based tools (`read`, `grep`, `find`, `edit`, `write`, and `lsp_*`) require an explicit `workdir`. Missing `workdir` is an error; do not rely on the session cwd. Use `.` only when the target root is the current working directory; otherwise use a relevant subdirectory or absolute directory.
- Use repository file tools for repository content work and reserve `bash` for command execution.
- Tool examples show only the arguments passed to the tool. Do not add wrapper metadata around the arguments.
- When `read` reports LSP support, use LSP tools for fast diagnostics, known-symbol navigation, and third-party API inspection when they fit the task. LSP tools require `path`, and that `path` is also used to infer the workspace root and select the correct server. For Java external definitions, prefer `lsp_workspace_symbols` or `lsp_goto_definition` to discover the class target, then pass that result to `lsp_java_decompile`.
- Few-shot examples:
  - Repository discovery: `find arguments {"pattern":"*.ts","path":"src","workdir":"packages/web"}` -> `grep arguments {"pattern":"createDemoDirectory","path":"src","workdir":"packages/web","literal":true}` -> `read arguments {"path":"src/example.ts","workdir":"packages/web","offset":40,"limit":20}`
  - Local edit from fresh anchors: `read arguments {"path":"src/example.ts","workdir":"services/api"}` -> `edit arguments {"path":"src/example.ts","workdir":"services/api","edits":[{"replace_lines":{"start_anchor":"45#2574","end_anchor":"45#2574","new_text":"export function buildDemoDirectory(): UserDirectory {"}}]}`
  - Parallel independent reads: `read arguments {"path":"src/a.ts","workdir":"packages/app"}` and `read arguments {"path":"src/b.ts","workdir":"packages/app"}` in parallel when the results do not depend on each other.
  - Java third-party inspection: `lsp_workspace_symbols arguments {"path":"src/main/java/com/acme/App.java","workdir":"services/java","query":"String","limit":20}` -> `lsp_java_decompile arguments {"path":"src/main/java/com/acme/App.java","workdir":"services/java","target":"jdt://contents/java.base/java/lang/String.class?..."}`
